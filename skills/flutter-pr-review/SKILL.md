# Flutter PR Review

Review Flutter PRs against Clean Architecture, BLoC state management, SOLID, DRY, critical bugs/memory leaks, performance and blast-radius risks, and PR completeness. Add concise, actionable comments (problem, reason, solution) with severity levels.

### AI attribution (mandatory on every comment)

**Every** review comment you draft, show in the pre-post summary, or post to GitHub (line-level PR comments, review body, PR-body gap notes) **must** end with this exact line on its own paragraph (after Problem / Why / Suggestion):

`🤖 Generated by AI, co-authored and refined by the developer.`

Do not omit this line for any comment type.

---

## Required rules — no delete actions

**MANDATORY: This skill must never run any delete command or action. Never delete anything.**

- **Terminal:** Do not run `git`, `gh`, or any CLI command that deletes or removes (e.g. `git branch -d`, `git push --delete`, `gh repo delete`, `rm -rf`).
- **GitHub:** Do not call any MCP or API that deletes branches, repos, comments, issues, or other resources. Do not approve or merge PRs unless the user explicitly asks to merge.
- **Jira:** Do not call any Jira MCP tool that deletes or removes issues.
- **Filesystem:** Do not delete files or directories as part of the review.
- **Scope:** Read-only and additive only: fetch PR data, read diffs, post review comments. If the user asks to delete something, do not do it; tell them they must perform the delete themselves and give exact steps or commands they can run.

---

## Tools

- **GitHub**: `gh` CLI (authorized). Use for PR details, diff, and posting comments.
- **Jira MCP**: `user-jira-mcp`. Use when the PR title/description references a Jira issue (e.g. REFFER-123) to fetch context via `getJiraIssue` if it helps the review.

## Process

1. **Resolve PR**
   - Input: PR number, PR URL, or branch name.
   - From repo root run: `gh pr view <pr> --json title,body,number,headRefOid,headRefName,baseRefName,files` and `gh pr diff <pr>` (or `--name-only` then full diff per file if needed).
   - Parse changed files; focus on `lib/` (and test files if present).
   - **Diff-side line mapping:** When parsing the unified diff, track old-file line numbers (lines prefixed with `-` or context lines) and new-file line numbers (lines prefixed with `+` or context lines) separately. For every issue you identify, determine whether the problematic code exists on the old side (removed/modified line) or the new side (added/unchanged line). This mapping is required for correct comment placement in step 6.

2. **Optional Jira context**
   - If PR mentions a Jira key (e.g. RT-123, REFFER-456), call Jira MCP `getJiraIssue` (with the project's `cloudId`) to get summary/description. Use only to align review with ticket intent, not to skip checks.

2b. **Branch-type review mode**
   - Use the `headRefName` obtained in step 1 to determine the branch name.
   - If the branch name starts with `fix/`, activate **bug-fix review mode**:
     - **Priority order:** Critical Issues (section 5) > DRY (section 4) > Performance & blast radius (section 6) > everything else.
     - **Suppress** Clean Architecture (section 2) and SOLID (section 3) comments unless the violation is directly causing the bug or introduces a new crash/regression.
     - **State Management (section 1)** comments are allowed only if the change introduces an actual bug (e.g., infinite rebuild, missing dispose).
     - The review goal is: "Does this fix solve the bug safely without introducing new issues or breaking other parts of the app?"
     - DRY should **always** be checked regardless of branch type.
   - If the branch does **not** start with `fix/`, run the full review with all six checklists as normal.

3. **Review each changed file**

   **Context-first review principle (mandatory):**
   - Before applying any checklist, first understand the **intent** of the PR from its title, description, Jira ticket, and the diff as a whole.
   - Evaluate every potential comment against this intent: does the issue directly affect the correctness, safety, or stated goal of this PR?
   - **Suppress** comments that are tangential to the PR's purpose unless they are **[Critical]** severity (bugs, crashes, memory leaks).
   - For `[Recommended]` and `[Low]` comments, only include them if they are directly within the changed lines or immediately affect the changed logic. Do not flag pre-existing patterns in unchanged code that the PR did not touch.

   - For each relevant file, run the applicable checklists below (State Management, Clean Architecture, SOLID, DRY, Critical Issues, Performance & blast radius) — subject to the branch-type review mode from step 2b.
   - Note bugs, bottlenecks, readability, and improvements. Record file path and line(s) for comments.

3a. **PR completeness check**
   - Read the PR title and body (and Jira ticket description if available) carefully.
   - List every fix, feature, or requirement stated in the description.
   - Map each stated item to the actual diff: confirm it is addressed by the changes.
   - Flag any stated item that is **not covered** by the diff as a gap comment on the PR body.
   - Also flag any significant change in the diff that is **not mentioned** in the description (undocumented side-effects).

4. **Write comments**
   - **Format per comment:**
     - **Severity Level** (in bold, at the very beginning): `**[Critical]**`, `**[High]**`, `**[Recommended]**`, or `**[Low]**`
     - **Problem:** One short sentence.
     - **Why it matters:** Max 2 lines.
     - **Suggestion:** Up to 3 lines of text or a small code block.
     - Then a blank line, then the mandatory AI attribution line (see "AI attribution" above).
   - Prefer line-level comments; use a single summary comment only when there's no specific line.
   
   **Severity Level Criteria:**
   - **[Critical]**: App-breaking bugs, security vulnerabilities, memory leaks, or issues that will cause a crash or regression in production.
   - **[High]**: Potential logic errors, performance bottlenecks (e.g., unnecessary rebuilds), or violations of the project's architectural patterns.
   - **[Recommended]**: Best practices for Flutter/Dart, improved readability, or minor optimizations that would improve the long-term health of the codebase.
   - **[Low]**: Minor style "nits," non-urgent documentation updates, or subjective naming preferences that don't impact functionality.

   **Scope gate for lower-severity comments:** When the PR is a targeted fix, raise the bar for `[Recommended]` and `[Low]` comments: only include them if they are on changed lines and relevant to the fix's correctness or immediate readability. Avoid recommending refactors that expand the scope of a focused PR.

5. **Show comment summary (before posting)**
   - **Required:** Before calling the GitHub API or `gh pr review`, output a clear list of every comment that will be added.
   - For each comment show: **file path**, **line number** (or "PR body" for a single overall comment), and the **full comment text** (Severity Level, Problem, Why, Suggestion, plus the mandatory AI attribution line).
   - Example format:
     ```
     Comment 1 — lib/feature/screen.dart (line 42)
     **[High]**
     Problem: ...
     Why: ...
     Suggestion: ...

     🤖 Generated by AI, co-authored and refined by the developer.

     Comment 2 — lib/feature/bloc.dart (line 15)
     **[Recommended]**
     ...
     🤖 Generated by AI, co-authored and refined by the developer.
     ```
   - Only after this summary is visible to the user, proceed to post comments (or ask for confirmation if you prefer).

6. **Post comments**
   - Each posted `body` must include the full formatted comment **including** the severity level at the beginning and the mandatory AI attribution line at the end.
   - Get latest commit on PR head: `gh pr view <pr> --json headRefOid -q .headRefOid`.
   - Use repo from `gh repo view --json nameWithOwner -q .nameWithOwner` if needed.

   **Diff-side targeting rule (mandatory):**
   - The comment **must** be placed on the line where the **problem exists**, not on the line where a fix would be inserted.
   - If the problematic code is a **removed line** (only in the old file / `-` side of the diff), use `side=LEFT` and set `line` to the old-file line number.
   - If the problematic code is an **added or unchanged line** (in the new file / `+` or context side), use `side=RIGHT` and set `line` to the new-file line number.
   - When your suggestion involves adding code at a different location than the problem, still anchor the comment on the problem line and describe the target location in the suggestion text.

   **Line number validation (mandatory):**
   - **All line parameters must be positive integers.** Never pass a range string (`42-50`), a float, a string, or an empty value to any line field.
   - Before constructing each `gh api` call, verify every line value is a bare integer. If it is not, fix it before posting.

   **Line-level comments — single-line and multi-line formats:**

   - **Single-line comment** (issue on one line):
     ```
     gh api repos/{owner}/{repo}/pulls/<pr>/comments \
       -f body="<escaped markdown>" -f commit_id="<headRefOid>" -f path="<file path>" \
       -f line=<integer> -f side=<LEFT|RIGHT>
     ```
   - **Multi-line comment** (issue spans several lines):
     ```
     gh api repos/{owner}/{repo}/pulls/<pr>/comments \
       -f body="<escaped markdown>" -f commit_id="<headRefOid>" -f path="<file path>" \
       -f start_line=<integer> -f start_side=<LEFT|RIGHT> \
       -f line=<integer> -f side=<LEFT|RIGHT>
     ```
     - `start_line` = the **first** (topmost) line of the range. `line` = the **last** (bottommost) line.
     - `start_line` must be less than or equal to `line`. Both must be integers.
     - `start_side` and `side` indicate the diff side for the start and end lines respectively.

   - **Single overall review body:** `gh pr review <pr> --comment -b "<body>"` or `-F-` with a file.
   - If posting many line comments, consider batching and respecting rate limits (short delay between requests if needed).
   - Do not post until the comment summary (step 5) has been shown.

## 1. State Management (BLoC over StatefulWidget)

- [ ] **StatefulWidget:** Any new `StatefulWidget`? Flag it. Allow only for local UI state (animations, scroll controllers, focus nodes). Otherwise prefer StatelessWidget + BLoC.
- [ ] **Business logic in UI:** Is business logic, API calls, or state transformation in the widget/file? Flag it. These must live in a `Bloc` or `Cubit`.
- [ ] **UI rebuilds:** Are `BlocBuilder` / `BlocListener` / `BlocConsumer` used so only affected widgets rebuild? Flag broad builders that could be replaced with `BlocSelector` for a smaller state slice.

## 2. Clean Architecture (Data, Domain, Presentation)

- [ ] **Layer separation:** Presentation only talks to BLoCs/Cubits. BLoCs only to UseCases (Domain). UseCases only to Repository interfaces (Domain). Implementations only in Data. Flag any cross-layer violation (e.g. UI calling a datasource, BLoC importing a Flutter widget).
- [ ] **Domain purity:** In `domain/` (entities, use cases, repository interfaces): no `import 'package:flutter/...'` and no third-party packages (e.g. HTTP, dio). Pure Dart only. Flag any Flutter or external import in Domain.

## 3. SOLID

- [ ] **SRP:** Does each class/widget/function have a single reason to change? If a widget both builds UI and formats dates (or does other distinct responsibilities), suggest splitting (e.g. formatter helper, smaller widgets).
- [ ] **DIP:** Are dependencies on concrete implementations instead of abstractions? Repositories and external services should be injected via interfaces/abstract classes, not `new MyRepo()` or global getters inside the class. Flag and suggest injecting the abstraction.

## 4. DRY

- [ ] **Repeated UI:** Same or very similar widget trees in multiple places? Suggest extracting a reusable stateless widget.
- [ ] **Repeated logic:** Same validation, date formatting, or error handling in more than one place? Suggest a shared utility, extension, or helper (in `core/` or appropriate layer).

## 5. Critical Issues (Bugs, Memory Leaks, and Safety)

- [ ] **StreamSubscription / listeners not cancelled:** Any `StreamSubscription`, `AnimationController`, `TextEditingController`, `ScrollController`, `FocusNode`, or similar resource that is created but not disposed in `dispose()`? Flag it as a memory leak.
- [ ] **BLoC / Cubit not closed:** Is a `Bloc` or `Cubit` instantiated manually (e.g. `MyBloc()`) without being provided via `BlocProvider` (which auto-closes)? Flag: must be closed or provided via `BlocProvider`.
- [ ] **async gaps without mounted check:** After any `await` inside a `State` method, is `if (!mounted) return;` missing before using `context` or calling `setState`? Flag it — this can crash after widget disposal.
- [ ] **Null safety violations:** Forced unwrap (`!`) on a value that could realistically be null at runtime? Flag and suggest null-safe alternatives.
- [ ] **Uncaught exceptions:** Is there a `Future` or `Stream` that can throw but has no error handling (`catchError` / `try-catch`)? Flag — silent failures make debugging hard.
- [ ] **Infinite rebuilds / setState in build:** Is `setState`, `add` (BLoC event), or `emit` called from within a `build` method or a `BlocBuilder` body? Flag — this creates infinite rebuild loops.
- [ ] **Heavy work on the UI thread:** Expensive synchronous operations (large loops, JSON parsing, image processing) done directly in `build` or event handlers without isolates? Flag and suggest moving to an isolate or background compute.
- [ ] **Race conditions:** Multiple async operations that mutate the same state without guards or cancellation tokens? Particularly in search/debounce patterns — flag if the latest result isn't guaranteed to win.
- [ ] **Global/static mutable state:** New `static` mutable variables or singletons introduced? Flag — these are hard to test and can cause unexpected cross-screen state sharing.
- [ ] **Platform channel / plugin misuse:** Are platform calls made on a non-main thread or without proper error handling? Flag and suggest wrapping in try-catch with a fallback.

## 6. Performance & blast radius

Apply [Flutter performance best practices](https://docs.flutter.dev/perf/best-practices) mentally: flag issues that waste frames, memory, or work; avoid nitpicks unless impact is plausible.

- [ ] **Rebuild / layout cost:** Wider than needed `BlocBuilder`/`setState` scope, removed or missing `const` where children are stable, or ancestor invalidations that rebuild large subtrees? Flag.
- [ ] **Lists & scrolling:** Long or dynamic lists built with default `ListView`/`GridView`/`Column` children instead of `.builder` / sliver builders; avoidable `shrinkWrap` or nested scrollables causing extra layout? Flag.
- [ ] **Pictures & decoding:** Large raster assets or network images without reasonable decode bounds (`cacheWidth` / `cacheHeight`, `ResizeImage`, or equivalent) when many instances or big sources are likely? Flag.
- [ ] **Expensive paint / effects:** New `BackdropFilter`, heavy clipping, or opacity on large areas without clear need; gratuitous `RepaintBoundary` (or obvious missing boundary only when profiling would justify it)? Flag proportionately.
- [ ] **Ripple / tangent impact:** Changes to shared widgets, utils, routes, DI, repository contracts, or defaults that other features rely on — could this break or slow unrelated screens, APIs, or navigation? Flag and suggest call-site review, tests, or release notes.

**StatefulWidget used for app state:**

**[High]**
- **Problem:** App state is held in a `StatefulWidget` with `setState`.
- **Why:** This couples UI and business logic and makes testing and reuse hard. We use BLoC for app/feature state.
- **Suggestion:** Move state and logic into a `Cubit`/`Bloc` and use `BlocProvider` + `BlocBuilder` in a `StatelessWidget`. Keep this widget only for layout.

**[High]**
- **Problem:** `domain` layer file imports `package:flutter/material.dart`.
- **Why:** Domain must stay framework-agnostic so business rules are testable and reusable.
- **Suggestion:** Remove Flutter from `domain/`. Move any UI-related logic to presentation or a mapper in `data/`/`presentation`.

**Duplicate logic:**

**[Recommended]**
- **Problem:** Date formatting string is duplicated in three places.
- **Why:** Changes (e.g. locale or format) would require edits in multiple files and can get out of sync.
- **Suggestion:** Add a single formatter in `core/utils` or an extension on `DateTime` and use it everywhere.

**Memory leak — controller not disposed:**

**[Critical]**
- **Problem:** `TextEditingController` is created in `initState` but never disposed.
- **Why:** The controller holds a reference to the widget's state; not disposing it leaks memory on every navigation.
- **Suggestion:** Override `dispose()` and call `_controller.dispose()` before `super.dispose()`.

🤖 Generated by AI, co-authored and refined by the developer.

**async gap without mounted check:**

**[Critical]**
- **Problem:** `context` is used after an `await` without checking `if (!mounted) return;`.
- **Why:** The widget may have been removed from the tree while the async operation was running, causing a crash.
- **Suggestion:** Add `if (!mounted) return;` immediately after each `await` before accessing `context` or calling `setState`.

**PR description gap:**

**[High]**
- **Problem:** The PR description states "[requirement X]" but no changes in the diff address it.
- **Why:** Reviewers and QA will assume the feature/fix is complete; missing coverage can ship a regression or incomplete feature.
- **Suggestion:** Either add the missing implementation or update the PR description to clarify it is out of scope for this PR.

**List performance:**

**[High]**
- **Problem:** A long list is built with `ListView(children: [...])` (or many `Column` children) instead of `ListView.builder`.
- **Why:** All children are built and laid out up front, which hurts scroll performance and memory as the list grows.
- **Suggestion:** Use `ListView.builder` (or `SliverChildBuilderDelegate`) so only visible items are built; keep item widgets `const` where possible.

## Checklist summary

Before finishing the review, confirm:

- [ ] All changed files under `lib/` (and relevant tests) were reviewed.
- [ ] Each comment has: severity level, problem, why (≤2 lines), suggestion (≤3 lines or small code block), plus the mandatory AI attribution line.
- [ ] **PR completeness checked:** every item in the PR description (and Jira ticket if present) was mapped to the diff; gaps and undocumented changes are flagged.
- [ ] **Critical issues checked:** memory leaks, async safety, null safety, infinite rebuilds, race conditions, and heavy UI-thread work were all inspected.
- [ ] **Performance & blast radius checked:** rebuild/list/image/paint risks and shared-API or cross-feature ripple effects from §6 were considered.
- [ ] Comment summary (file, line, full text for each) was shown to the user before posting.
- [ ] Comments were posted via `gh` (line-level and/or review body) or clearly listed for the user to paste.
- [ ] **Branch-type mode applied:** If the branch starts with `fix/`, confirmed that Clean Architecture and SOLID comments were suppressed unless they directly cause a bug or regression. DRY was still checked.
- [ ] **No delete actions:** No delete/destructive commands or MCP actions were run (see "Required rules — no delete actions" above).
